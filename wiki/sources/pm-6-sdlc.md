**Главная идея:** PM должен понимать технические основы SDLC, модели разработки и инструменты команды (Git, CI/CD, DevOps), чтобы эффективно планировать, оценивать риски и говорить с разработчиками на одном языке.

## Ключевые темы лекции

### SDLC и его этапы
SDLC — условная схема создания ПО. Стандарты: ГОСТ 34.601-90 и ISO/IEC 12207:2008. Шесть этапов: планирование и анализ требований → проектирование архитектуры → разработка → тестирование → развёртывание → сопровождение. Подробнее: [[sdlc]].

### Артефакты: SRS и DDS
- **SRS** (Software Requirements Specification) — фиксирует все требования по итогам этапа планирования; фундамент для всех последующих этапов и для критериев качества в тестировании.
- **DDS** (Detail Design Specification) — описывает модули, связи между ними и потоки данных (Data Flow Diagram); общий язык команды на этапе проектирования.

### Модели SDLC
Рассмотрено пять моделей: Waterfall, итеративно-инкрементальная, спиральная, V-модель, RAD. Подробнее: [[sdlc]], [[waterfall]].

### Jira workflow как отражение SDLC
Статусы задачи (Open → In BA Review → In Tech Review → Ready for Dev → In Progress → Dev Complete → Ready for Test → In Test → Bug Fixing → Closed / At Risk) в точности отображают этапы SDLC. Это позволяет PM читать состояние проекта через доску.

### DevOps
Культура на стыке Dev и Ops. 8-этапный бесконечный цикл. Инструменты по каждому этапу. Подробнее: [[devops]].

### Build vs Release
- **Build** — собранное приложение, переданное команде тестирования.
- **Release** — официальный выпуск для конечных пользователей после сертификации build командой QA.
Не каждый build становится release. Подробнее: [[devops]].

### Deployment Management vs Release Management
- **Deployment Management** — техническая часть: перенос, установка, конфигурация компонентов в целевую среду.
- **Release Management** — весь процесс от Request for Changes до Issue Reporting & Collection (8 этапов цикла). Подробнее: [[devops]].

### Software Environments
Development → Staging/PreProd → Production. Среды изолируют этапы работы и снижают риск «протечки» нестабильного кода к пользователям. Подробнее: [[devops]].

## Практические задания
7 заданий для очного занятия:
1. Выбор модели SDLC для двух контрастных проектов (MVP стартапа vs B2B-интеграция с фиксированным ТЗ).
2. Составление минимального SRS и структуры DDS по голосовому сообщению заказчика.
3. Диагностика задачи, застрявшей в статусе Reopened в Jira.
4. Объяснение заказчику разницы Build vs Release по реальной ситуации с 5 build-ами.
5. Проектирование процесса Release Management при 15 запросах и возможности реализовать 4 фичи.
6. Ситуация конфликта разработчика и ревьюера в Pull Request.
7. Выбор между Continuous Delivery и Continuous Deployment для финтех-стартапа с регуляторными ограничениями.
