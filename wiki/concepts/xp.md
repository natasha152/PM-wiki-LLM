# XP (Extreme Programming)

Из [[pm-4-agile-methodologies]].

Гибкая методология, применяемая **исключительно в разработке ПО** (в отличие от [[scrum]] или [[kanban]], которые применимы и за пределами IT). Создана Кентом Беком, чтобы быстро адаптироваться к меняющимся требованиям и повысить качество кода через сильное взаимодействие в команде.

## Четыре ключевых процесса
Кодирование по единому стандарту, тестирование (включая **TDD** — тесты пишутся до кода), чёткое планирование (долгосрочные цели релиза + короткие итерации, обычно раз в две недели), постоянное слушание (диалог внутри команды и с клиентом).

## Ключевые практики
- **TDD (Test-Driven Development)** — сначала пишется тест (изначально не проходит), затем реализуется логика для его прохождения; включает unit-тесты (отдельные модули) и функциональные тесты (логика взаимодействия компонентов).
- **Парное программирование** — двое программистов за одним компьютером (один пишет, другой следит за общей картиной), пары регулярно меняются — снижает bus factor, повышает качество кода за счёт немедленного обнаружения ошибок.
- **Continuous Integration** — код интегрируется несколько раз в день, а не собирается в конце проекта.
- **Refactoring** — постоянное улучшение структуры кода без изменения функциональности; безопасность обеспечивают автоматические тесты.
- **Small Releases** — частые небольшие релизы с минимальным, но полезным функционалом.
- **Simple Design** — минимализм в проектировании: решаем только текущие задачи, дизайн адаптируется по мере роста требований.
- **Коллективное владение кодом** — любой разработчик может менять любой участок системы; снижает bus factor.
- **Единые стандарты оформления кода**.
- **Planning Game** — динамичное планирование с user stories (карточки) и story points для оценки сложности; заказчик отвечает за бизнес-решения, разработчики — за технические.

Заказчик в XP — это конечный пользователь продукта, постоянно доступный для команды (не тот, кто платит за проект).

## XP vs Scrum vs Kanban vs Lean
- **vs Scrum** — у XP короче итерации и допускаются изменения внутри итерации (Scrum строго держит цель и Sprint Backlog); XP фокусируется на технических практиках, Scrum — на управлении проектом, ролях и процессах; в XP вектор задаёт постоянно доступный клиент, в Scrum — самоорганизующаяся команда с PO.
- **vs Kanban** — XP основан на коротких итерациях с чёткими целями и ролями, Kanban — на непрерывном потоке без фиксированных сроков и без жёстко закреплённых ролей; XP делает упор на технические практики, Kanban — на визуализацию потока (WIP-лимиты).
- **vs Lean** — XP работает итерациями, завершающимися релизом полезного функционала, Lean ориентирован на MVP с минимальными затратами; XP — про технические практики, Lean — про оптимизацию процесса и снижение потерь (см. [[lean]]).

XP может комбинироваться с другими методологиями на одном проекте (например, в Team Level [[scaling-frameworks|SAFe]] упоминаются и Scrum-, и XP-команды).
