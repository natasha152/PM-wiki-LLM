# SDLC — Software Development Life Cycle

Из [[pm-6-sdlc]].

**SDLC** — условная схема (жизненный цикл), включающая этапы создания программного обеспечения. На каждом этапе выполняются свои задачи, в итоге продукт должен быть качественным и отвечать заявленным требованиям.

Регламентирующие стандарты: **ГОСТ 34.601-90** и **ISO/IEC 12207:2008** Systems and Software Engineering.

---

## Этапы SDLC

| # | Этап | Ключевой результат |
|---|------|-------------------|
| 1 | **Planning & Requirements** — сбор бизнес- и технических требований, ТЭО, оценка рисков | SRS (Software Requirements Specification) |
| 2 | **Design** — проектирование архитектуры, выбор оптимального варианта дизайна | DDS (Detail Design Specification) с диаграммами модулей и Data Flow Diagrams |
| 3 | **Development** — написание кода по DDS и SRS, выбор языка/библиотек/инструментов | Собранное приложение (build) |
| 4 | **Testing** — выявление, фиксация, исправление дефектов до соответствия стандартам из SRS | Сертифицированный build |
| 5 | **Deployment** — выпуск продукта (поэтапно: UAT → 5% пользователей → полный релиз) | Release |
| 6 | **Maintenance** — исправление ошибок конечных пользователей, улучшения | Поддерживаемый продукт |

> Тестирование применяется на каждом этапе, но на этапе 4 становится основным фокусом.

## Модели SDLC

| Модель | Суть | Когда применять |
|--------|------|-----------------|
| **Waterfall** | Линейно-последовательная, фазы не перекрываются | Стабильные требования, Fixed Price контракт — см. [[waterfall]] |
| **Итеративно-инкрементальная** | Новая версия ПО в конце каждой итерации; комбинация инкремента (добавление частей) и итерации (улучшение всего) | Меняющиеся требования, MVP — см. [[agile]] |
| **Спиральная** | Гибрид итерационного подхода и анализа рисков | Высокая неопределённость, сложные проекты |
| **V-модель** | Каждый этап разработки имеет зеркальный этап тестирования (форма буквы V) | Критичные системы, где тестирование неотделимо от разработки |
| **RAD** (Rapid Application Development) | Акцент на скорость, готовые прототипы, минимальное планирование | Сжатые сроки, прототипирование |

На практике чаще всего используются Waterfall и итеративно-инкрементальная.

**Инкремент vs итерация** (аналогия с картиной):
- Инкремент — рисуем по частям: сначала левая часть, затем правая (добавляем новые функции порциями).
- Итерация — рисуем набросок всей картины, потом уточняем детали (улучшаем весь продукт).

## SDLC и Jira Workflow

Жизненный путь задачи в Jira отражает этапы SDLC:

```
Open → In BA Review → In Tech Review → Ready for Dev
     → In Progress → Dev Complete → Ready for Test
     → In Test ↔ Bug Fixing → Closed
                               At Risk (кросс-этапный статус)
```

Этот параллелизм позволяет PM читать состояние проекта через доску задач.
