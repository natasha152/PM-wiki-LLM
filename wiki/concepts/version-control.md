# Version Control / Git / GitHub

Из [[pm-6-sdlc]].

---

## Version Control Systems (VCS)

Система контроля версий отслеживает изменения в коде и позволяет нескольким разработчикам работать параллельно без конфликтов. Это основа Code-этапа в [[devops]]-цикле.

---

## Git

Распределённая система контроля версий. Преимущества: быстрая, бесплатная, локальные операции, полная история изменений, простое ветвление.

**Ключевые понятия:**

| Понятие | Определение |
|---------|------------|
| **Commit** | Фиксация изменений в репозитории. Атрибуты: имя, дата, автор, комментарий. |
| **Branch** | Указатель на commit с определёнными изменениями. Ветвление позволяет работать в изоляции от основной линии. |
| **main / master** | Основная ветка проекта; разработчики создают feature-ветки на её основе. |
| **Merge** | Слияние ветки в другую (обычно в main). |

---

## GitHub

Веб-сервис для хостинга Git-репозиториев с дополнительными функциями: issue tracking, wiki, web-интерфейс для просмотра кода.

Проект хранится в **репозитории** — коллекции всех изменений кода. Каждый разработчик клонирует репозиторий и работает локально.

---

## Pull Request (PR)

**Pull Request** — заявка на слияние кода из feature-ветки в основную. В процессе PR Git показывает diff: добавленные строки (зелёным) и удалённые (красным).

Перед финальным merge все разработчики должны просмотреть изменения (**Code Review**) и принять их (**Approve**).

**Зачем нужен Code Review:**
- Контроль качества до попадания кода в основную ветку.
- Выявление скрытых уязвимостей и архитектурных проблем.
- Обмен знаниями внутри команды.
- Прозрачность изменений для PM и команды.

> Merge без Approve — риск для проекта даже при наличии автоматических тестов в CI: тесты проверяют корректность, но не архитектурные и security-проблемы.
