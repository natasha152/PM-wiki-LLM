# Kanban

Из [[pm-4-agile-methodologies]].

Метод управления потоком задач, родившийся на заводах Toyota в 1950-х (мастера вывешивали списки задач на видных местах). В отличие от [[scrum]] не работает спринтами — задачи поступают и выполняются непрерывным потоком.

## Четыре принципа
- **Точно в срок** — производить/разрабатывать ровно столько, сколько нужно, и ровно тогда, когда нужно.
- **Наглядность** — весь процесс отображается на доске (физической или цифровой — Trello, Jira, Asana); виден статус каждой задачи и узкие места.
- **Равномерная нагрузка** — задачи берутся по мере готовности, без перегрузки команды.
- **Постоянное улучшение** — застрявшая на этапе задача — сигнал для немедленного изменения процесса.

## Kanban-доска и WIP-лимиты
Доска разделена на колонки-этапы (например: Backlog → Разработка → Тестирование → Релиз); чем меньше колонок, тем проще отслеживать прогресс. Каждая карточка — задача с кратким описанием, весом (час/день, оценивает сама команда) и приоритетом.

Каждая колонка имеет **WIP-лимит** (ограничение числа задач одновременно). Если колонка заполнена — новые задачи туда не берутся, даже если они есть в Backlog. Цель — не максимизировать число задач в работе, а **ускорить равномерное прохождение** каждой задачи через все этапы. При обнаружении узкого места (например, колонка «Тестирование» переполнена) команда должна гибко перераспределить ресурсы — например, разработчики подключаются к тестированию вместо того, чтобы брать новую работу и забивать свою колонку ещё больше.

## Когда выбирать Kanban
- Работа делится на небольшие независимые задачи/запросы.
- Команда занимается багфиксом, поддержкой продукта, небольшими улучшениями (саппорт).
- Уже налажен процесс быстрой регулярной поставки (CI/CD).
- Нет жёстких дедлайнов для крупных поставок — работа в удобном темпе с фокусом на качество, а не на сроки.
