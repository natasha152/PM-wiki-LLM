# ЛЕКЦИЯ ПО ПРОЕКТНОМУ МЕНЕДЖМЕНТУ

**Тема:** Жизненный цикл разработки программного обеспечения (SDLC), модели SDLC и связанные технические концепции (DevOps, Git, CI/CD)

**Главная идея:** Проектный менеджер должен понимать технические основы SDLC, модели разработки и инструменты команды (Git, CI/CD, DevOps), чтобы эффективно планировать работу, оценивать риски и говорить с разработчиками на одном языке.

---

## Основные тезисы

1. **SDLC и его этапы**
   SDLC — это условная схема создания ПО, регламентированная стандартами ГОСТ 34.601-90 и ISO 12207. Основные этапы: планирование и анализ требований (результат — SRS), проектирование архитектуры (результат — DDS), разработка, тестирование, развёртывание и сопровождение (maintenance).

2. **Документы-артефакты SRS и DDS**
   SRS (Software Requirements Specification) фиксирует требования к продукту и служит основой для всех последующих этапов. DDS (Detail Design Specification) описывает модули, связи между ними и потоки данных (диаграммы модулей, Data Flow Diagram) — общий язык для команды.

3. **Поэтапное развёртывание и UAT**
   Развёртывание часто происходит постепенно (например, на 5% пользователей) с проведением User Acceptance Testing для сбора обратной связи перед полным релизом.

4. **Модели SDLC: водопад, итеративная, спиральная, V-модель, RAD**
   Каждая модель имеет особенности и область применения: водопад — линейный и предсказуемый, но негибкий; итеративно-инкрементальная — гибкая, подходит для меняющихся требований; спиральная — для проектов с высокой неопределённостью; V-модель — связывает разработку и тестирование; RAD — акцент на скорость и прототипирование.

5. **Модель водопада: риски и снижение рисков**
   Жёсткая последовательность этапов без возврата назад делает модель рискованной для крупных проектов с долгим циклом. Снижение риска — модульная поставка (разделение на независимые части) и раннее вовлечение QA.

6. **Итеративно-инкрементальная модель**
   Инкремент — добавление новых функций по частям (аналогия — рисование картины по частям). Итерация — постепенное улучшение всего продукта (аналогия — рисование от наброска к деталям). На практике эти подходы комбинируются.

7. **Параллели SDLC и workflow задач в Jira**
   Статусы задач (To Do, In Progress, In Review, Ready for Test, In Test, Bug Fixing, Closed, At Risk) отражают этапы SDLC — от планирования до сопровождения. Это помогает менеджеру читать состояние проекта через доску задач.

8. **DevOps как подход и культура**
   DevOps объединяет разработку и эксплуатацию в непрерывный цикл из 8 этапов: Plan, Code, Build, Test, Release, Deploy, Operate, Monitor — с конкретными инструментами на каждом (Jira/Trello, Git/GitHub, Jenkins/Maven, Selenium/JUnit, GitHub Actions, Docker/Kubernetes, Grafana/Prometheus). Цель — устранить барьер между командами и ускорить безопасную доставку кода.

9. **Build vs Release**
   Build — собранная версия приложения для тестирования; Release — официальная версия, прошедшая тестирование и сертификацию, переданная конечным пользователям. Не каждый build становится релизом.

10. **Release Management vs Deployment Management**
    Deployment Management — техническая часть (перенос, установка, конфигурация компонентов). Release Management — более широкий процесс (запрос на изменения → планирование релиза → build → review → тестирование → deployment → support → сбор отчётности), который обеспечивает готовность сервиса к публичному использованию.

11. **Software Environment: Dev / Staging / Production**
    Среды изолируют этапы работы: Development (эксперименты разработчиков), Staging/Preprod (максимально похожа на production, для финальной проверки), Production (рабочая среда для конечных пользователей).

12. **Version Control и Git**
    Система контроля версий отслеживает изменения и позволяет командной работе без конфликтов. Git — распределённая, быстрая, бесплатная система с локальными операциями, полной историей и простым ветвлением. Ключевые понятия: commit (фиксация изменений), branch (ветка), merge (слияние).

13. **GitHub, Pull Request и Code Review**
    GitHub — сервис хостинга на базе Git с дополнительными функциями (issue tracking, wiki, web-интерфейс). Pull Request — запрос на слияние веток с возможностью обсуждения и code review до merge. Это обеспечивает прозрачность, контроль качества и обмен знаниями в команде.

14. **CI / CD: Continuous Integration, Delivery, Deployment**
    CI — автоматическая сборка и тестирование кода при каждом изменении. Continuous Delivery — автоматическая подготовка к деплою на production с ручным подтверждением финального шага. Continuous Deployment — полностью автоматическое развёртывание в production без ручного утверждения.

## Практические задания для очного занятия

### Задание 1: Выбор модели SDLC для проекта
**Контекст:** Стартап разрабатывает MVP мобильного приложения для бронирования услуг. Бюджет ограничен, требования могут измениться после первых отзывов пользователей. Параллельно крупный B2B-клиент заказывает интеграцию с фиксированным ТЗ и фиксированным бюджетом на 6 месяцев.
**Задача:** Студенты должны обосновать, какую модель SDLC (водопад, итеративно-инкрементальную, спиральную, V-модель или RAD) выбрать для каждого из двух проектов и объяснить риски выбранного подхода.
**Связь с темой лекции:** Закрепляет понимание особенностей и областей применения моделей SDLC.

### Задание 2: Восстановление SRS и DDS по обрывочным требованиям
**Контекст:** Product Manager получил от заказчика голосовое сообщение с пожеланиями к новому разделу личного кабинета: "хочу, чтобы пользователь видел историю заказов, мог их фильтровать и скачивать отчёт в PDF".
**Задача:** Составить минимальный план SRS-документа (функциональные/нефункциональные требования) и описать, какие элементы должны войти в DDS (модули, потоки данных) для передачи команде разработки.
**Связь с темой лекции:** Закрепляет понимание роли и содержания документов SRS и DDS на этапах планирования и проектирования.

### Задание 3: Чтение workflow задачи в Jira
**Контекст:** Менеджер смотрит на доску Jira и видит, что задача "Добавить экспорт в Excel" уже неделю находится в статусе Reopened, хотя разработка была завершена.
**Задача:** Определить, на каком этапе SDLC возникла проблема, какие вопросы задать команде и какие статусы должна пройти задача дальше, прежде чем перейти в Closed.
**Связь с темой лекции:** Закрепляет соответствие между статусами Jira и этапами SDLC, в частности роль этапа тестирования.

### Задание 4: Build, который не стал Release
**Контекст:** Команда за два дня собрала 5 build-ов новой версии приложения. Тестировщики нашли критичный баг в оплате в build №3, мелкие баги в build №4, а build №5 прошёл все тесты.
**Задача:** Объяснить заказчику разницу между build и release в данной ситуации и составить краткий чек-лист критериев, по которым build может стать release.
**Связь с темой лекции:** Закрепляет понятия Build, Release и роль QA в Release Management.

### Задание 5: Планирование Release Management
**Контекст:** В продукт поступило 15 запросов на новые функции от пользователей и стейкхолдеров за месяц. Команда может реализовать максимум 4 функции за один релизный цикл.
**Задача:** Спроектировать процесс Release Management для этого случая: как приоритизировать запросы, спланировать релиз с контрольными точками (разработка/тестирование/буфер), и что должно произойти после деплоя (сбор фидбека).
**Связь с темой лекции:** Закрепляет полный цикл Release Management — от Request for Changes до Reporting Collection.

### Задание 6: Pull Request и Code Review в конфликтной ситуации
**Контекст:** Два разработчика одновременно работали над веткой feature/login. Один открыл Pull Request, но ревьюер нашёл в коде потенциальную уязвимость и оставил комментарии. Разработчик не согласен с замечаниями и хочет сразу замержить ветку.
**Задача:** Описать, как должен действовать менеджер/тимлид в этой ситуации, какую роль играет Code Review, и почему merge без approve — это риск для проекта.
**Связь с темой лекции:** Закрепляет понимание процессов Pull Request, Code Review, Approve и Merge в Git.

### Задание 7: Выбор между Continuous Delivery и Continuous Deployment
**Контекст:** Финтех-стартап хочет ускорить выпуск обновлений, но юридический отдел требует обязательного ручного утверждения перед каждым релизом в production из-за регуляторных рисков.
**Задача:** Определить, какая практика подходит компании — Continuous Delivery или Continuous Deployment, и обосновать выбор с точки зрения автоматизации и контроля.
**Связь с темой лекции:** Закрепляет различие между CI, Continuous Delivery и Continuous Deployment.
